要想了解Kubernetes的volume，就必须先了解docker的存储。


Image and layers

建议先看下面的图来理解

  • 一个Docker镜像由多个层(Layer)组成。
  • 每个Layer代表Dockerfile中的一个指令。
  • 只有最后一层是可写的，其他Layer都是只读的。

例如：

FROM ubuntu:18.04
COPY . /app
RUN make /app
CMD python /app/app.py

上面的Dockerfile包括四个指令，具体含义很容易理解。每条指令都会创建一个Layer

  • 每个Layer都只描述了与上一层不同的部分。

  • 所有的Layer从上往下堆叠；

  • 当创建一个新的container，只是在所有Layer上面又加了一层新的Layer，且这层Layer是可写的。这个Layer叫做container layer。

  • 所有的写操作(增、删、改)都只会在这一层生效。



Storage Driver就是处理这些Layer之间的互相交互。

不同的storage driver各有优缺点、适用于不同的场景。



Container and Layers

由上一节可知，container和image之间最大的区别就是最上面的一层container layer(writable)。所有对容器的写操作都发生在这一层，当容器被删除时，这一层一起被删除，原来的镜像数据没有改变。


由于每个容器都有自己单独的container layer，所以即使多个容器由同一个镜像创建，每个容器都有自己的状态：

Docker使用storage driver来管理这些层(image layer / container layer)。每种storage driver实现方式不同，但都是基于copy on write(CoW)策略。


Container size on disk

可以使用docker ps -s命令来查看容器占用磁盘空间大小，有两个参数：

  • size: writeable layer占用磁盘大小。
  • virtual size: 镜像大小 + writeable layer size .

例如mgdb这个容器，406M(virtual size) - 38M(writeable layer size) = 368M(镜像大小)


CoW(Copy on write)

  • 如果一个文件在下面的layer，则上面的layer读取这个文件时，直接读取这个文件。
  • 当上面的layer需要更改这个文件时，这个文件就被复制到上面的layer。
  • 优点：可以减少I/O和每层的大小。另外，可以大大加速容器的启动时间，因为docker只需要创建writable container layer，这一层通常都不会太大。

但对于会写大量数据的容器，这个writable layer占用空间也会慢慢增大。

对于write-heavy型的应用，不应该将数据存储到容器内部，而是应当使用volume，这样可以提高I/O效率；而且，使用volume可以在不同容器之间共用。



不同的storage driver

Docker里的storage包括下面这些：

  • AUFS
  • Device Mapper
  • OverlayFS
  • ZFS
  • VFS
  • Brtfs

检查不同的layer

进入/var/lib/docker目录，可以看到overlay2文件夹。这个文件夹下面是各种layer的物理存储：


进入到任意一个layer文件夹下面，一般有以下内容：

[root@ip-172-31-30-76 fb11f74a5cec70dd30a23d4cb53c36814dff7222fb0fcf2a3928e65410bae806-init]# ls
diff  link  lower  work

  • diff文件夹是layer存储的实际文件（如果此layer不是最底层，则里面为copy on write时复制的文件）

  • link是该layer的软链。

  • lower用于docker组织layer的顺序，可以理解为排序使用。
[root@ip-172-31-30-76 fb11f74a5cec70dd30a23d4cb53c36814dff7222fb0fcf2a3928e65410bae806-init]# cat lower
l/A3Z2X6LM6HO3UYXSYROZXXG62L:l/F4KKP2ME4MXIOTBVQNQJIUX6TY:l/2H5THSGFFHFC5AIWTXU7PZLQMP
